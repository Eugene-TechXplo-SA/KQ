# Skill: 仕様書レビュー (spec-review)

## トリガー

「仕様書レビューして」「設計レビュー」「XXとYYの矛盾を調べて」など。

## 絶対ルール

- `docs/` は読み取り専用。絶対に編集しない。
- 修正案は `openspec/changes/` に出力する。
- Notion への反映はユーザー承認後のみ。

## 実行手順

### Step 1: レビュー対象を特定

ユーザーの指示から `docs/` 内の対象ファイルを特定する。
曖昧な場合はユーザーに確認する。

### Step 2: Agent Team を作成して並列レビューを実行

**必ず Agent Team を作成すること。Task（逐次実行）は使わない。**

以下のプロンプトで Agent Team を立ち上げる:

---

#### パターンA: ユーザーが観点を指定した場合

ユーザーの指示に合わせてチーム構成を組み替える。

例: 「仕様書とドメインモデルに矛盾がないか」

```
Agent Team を作成して。対象: [ファイルパス]

1. 仕様書アナリスト:
   - 仕様書を読み、全要件を構造化リストとして整理せよ
   - 曖昧な要件にはフラグを立てよ

2. ドメインモデルアナリスト:
   - ドメインモデルを読み、エンティティ・リレーション・制約を整理せよ
   - 仕様書アナリストの整理結果と突合し、矛盾を洗い出せ

3. 矛盾検出・修正提案者:
   - 両アナリストの成果を読み、矛盾・不整合のリストを作成
   - 各矛盾に対して修正案を提示
   - 深刻度を「致命的 🔴」「重大 🟡」「軽微 🟢」で分類

全員で議論し、最終的な矛盾リストと修正提案を合意してまとめよ。
出力: openspec/changes/[review-YYYYMMDD-テーマ]/review.md
```

---

#### パターンB: ユーザーが観点を指定しない場合（デフォルト）

```
Agent Team を作成して。対象: [ファイルパス]

1. 攻撃的レビュアー:
   - 対象ドキュメントの問題点を必ず5つ以上挙げよ
   - 設計者に一切同調するな。褒め言葉は禁止。
   - 観点: セキュリティ、スケーラビリティ、エッジケース、曖昧な記述、情報の欠落
   - 深刻度を「致命的 🔴」「重大 🟡」「軽微 🟢」で分類

2. 実現可能性レビュアー:
   - 技術的に実装可能かを検証せよ
   - コスト・工数・依存関係のリスクを評価せよ
   - 実装不可能な箇所には代替案を提示せよ

3. 統合・修正提案者:
   - 上記2名の指摘を読み、各指摘の妥当性を判定せよ
   - 正当な指摘 → 具体的な修正案を作成
   - 的外れな指摘 → 理由付きで却下
   - 最終的な修正提案をまとめよ

全員で議論し、最終合意した修正提案をまとめよ。
出力: openspec/changes/[review-YYYYMMDD-テーマ]/review.md
```

---

### Step 3: 修正案をファイルに保存

Agent Team の最終合意を以下に保存:

```
openspec/changes/[review-YYYYMMDD-テーマ]/
├── review.md      ← 指摘事項（深刻度付き）
└── proposal.md    ← 修正提案
```

### Step 4: ユーザーに確認を求める

```
📋 レビュー完了。確認してください:
  - openspec/changes/[変更名]/review.md
  - openspec/changes/[変更名]/proposal.md

→ 「承認」: Notion反映用のサマリーを出力します
→ 修正依頼: 具体的に指示してください
→ 「却下」: 破棄します
```
