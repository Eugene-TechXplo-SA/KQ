# 仕様書テンプレートガイド

> 本ドキュメントは、KING and QUEENプロジェクトの仕様書作成パターンを他プロジェクトに転用するためのテンプレートガイドである。

---

## 全体方針

### 仕様書の位置づけ

- **最上位層**（ドメインモデル + 仕様書）
- FE設計 / BE設計に渡すための**共通言語**
- 「何を実現するか」「どう振る舞うべきか」を定義する
- 実装詳細（UIコンポーネント、DB詳細、API Schema詳細）は**含めない**

### 言語・表記ルール

- **日本語**を主言語、英語ラベルを併記
- テーブルのヘッダーは日本語で統一し、English列を別途追加
- 技術用語・状態値は英語（例：`PENDING`, `APPROVED`）
- ID体系はプレフィックス + 連番（例：`BR-01`, `AC-01`, `TBD-01`）

### 相互参照ルール

| プレフィックス | 対象 | 例 |
|---|---|---|
| BR-XX | ビジネスルール | BR-01 口座連携の還元対象化 |
| AC-XX | 受入基準 | AC-01 口座連携 |
| TBD-XX | 未確定事項 | TBD-02 token_decimals |
| D-XX | 決定ログ | D-01 付与計算モデル |
| UC-XX | ユースケース | UC-01 口座連携申請 |
| 機能ID | 機能一覧の項目 | AUTH-01, ACC-01, RWD-01 |

## 作成の順序（推奨）

1. **01_purpose.md** — プロジェクトの「なぜ」を固める
2. **02_scope.md** — Phase1でやること/やらないことを線引き
3. **04_glossary.md** — チーム全員の共通言語を確立
4. **03_features.md** — 機能を洗い出して優先度付け
5. **05_usecases.md** — 主要フローを詳細化
6. **06_logic.md** — ビジネスルールを形式化
7. **07_state_transitions.md** — 状態遷移を図式化
8. **08_roles_permissions.md** — 誰が何をできるかを明確化
9. **09_acceptance_criteria.md** — 完成判定基準をGiven-When-Thenで
10. **11_external_interface.md** — API共通仕様を定義
11. **10_open_issues.md** — 未確定事項を集約管理
12. **画面設計書** — Figma等を元に画面単位で作成

## 品質基準

### 仕様書が満たすべき条件

- [ ] 全ドメイン用語が用語定義に登録されている
- [ ] 全機能にID（プレフィックス+連番）が振られている
- [ ] 主要フローにユースケースが書かれている
- [ ] 状態を持つエンティティに状態遷移図がある
- [ ] ビジネスルールが形式化されている
- [ ] 受入基準がGiven-When-Then形式で書かれている
- [ ] 未確定事項が影響度付きで集約されている
- [ ] Phase1/Phase2の境界が明確である

### リリース不可（NG）の条件

- 状態遷移の破綻
- 二重処理の可能性が残る
- 権限制約が守られない
- 追跡不能（説明可能性が担保できない）
- 緊急停止が機能しない
